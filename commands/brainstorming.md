# 头脑风暴 — 把想法变成设计

通过自然对话，把模糊的想法变成完整的设计方案。

先理解项目上下文，然后逐个问题澄清需求，理解清楚后提出设计方案并获得用户认可。

<HARD-GATE>
在你提出设计方案并获得用户批准之前，禁止调用任何实现类 skill、写任何代码、创建任何项目脚手架。
无论项目看起来多简单，都必须先过设计关。
</HARD-GATE>

## 反模式："这太简单了，不需要设计"

每个项目都要过这个流程。一个 todo list、一个工具函数、一个配置改动——全都要。
"简单" 的项目恰恰是未经审视的假设造成最多浪费的地方。
设计可以很短（真正简单的项目几句话即可），但必须提出来让用户确认。

## 检查清单

按顺序完成以下步骤：

1. **探索项目上下文** — 查看文件、文档、最近提交
2. **逐个澄清问题** — 一次一个问题，理解目的/约束/成功标准
3. **提出 2-3 个方案** — 附带权衡分析和你的推荐
4. **呈现设计** — 按复杂度分节讲解，每节确认后再继续
5. **写设计文档** — 保存并提交
6. **设计自审** — 检查占位符、矛盾、歧义、范围
7. **用户审阅** — 让用户确认设计文档后再继续
8. **过渡到实现** — 调用 `yeaft:writing-plans` skill 创建实现计划

## 流程

**理解想法：**

- 先查看当前项目状态（文件、文档、最近提交）
- 评估范围：如果需求涉及多个独立子系统，先拆分，再逐个 brainstorm
- 每次只问一个问题
- 优先用选择题（比开放问题更容易回答）
- 专注于：目的、约束、成功标准

**探索方案：**

- 提出 2-3 个不同方案及权衡
- 先推荐你认为最好的方案，解释理由
- 对话式呈现，不要堆砌

**呈现设计：**

- 确认你理解了要构建什么后，呈现设计
- 每节按复杂度决定篇幅——简单的几句话，复杂的 200-300 字
- 每节呈现后确认用户是否认可
- 覆盖：架构、组件、数据流、错误处理、测试
- 随时准备返回澄清

**在现有代码库中：**

- 先探索现有结构再提议改动，遵循既有模式
- 如果现有代码的问题影响当前工作，可以在设计中包含针对性改进
- 不做无关的重构

## 多角色 Brainstorm 模式（可选）

当问题需要多角度视角时，dispatch 并行 subagent，每个带不同 persona 独立思考：

### 触发条件
用户说 "我想要多角度视角" 或问题跨领域（产品 + 工程 + 设计）。

### 流程

1. 清晰陈述问题
2. dispatch 2-3 个 subagent **并行**执行，每个带不同 persona：

   **Subagent A（Jobs 视角）：**
   ```
   Task tool:
     prompt: |
       <persona>[yeaft:persona-pm-jobs 的内容]</persona>
       从你的角度 brainstorm 这个问题：
       [问题描述]
       输出：3-5 个关键观点 + 你的建议
       不要考虑其他角度——专注于你的视角。
   ```

   **Subagent B（Torvalds 视角）：**
   ```
   Task tool:
     prompt: |
       <persona>[yeaft:persona-dev-torvalds 的内容]</persona>
       从你的角度 brainstorm 这个问题：
       [问题描述]
       输出：3-5 个关键观点 + 你的建议
       不要考虑其他角度。
   ```

   **Subagent C（Rams 视角）：**
   ```
   Task tool:
     prompt: |
       <persona>[yeaft:persona-designer-rams 的内容]</persona>
       从你的角度 brainstorm 这个问题：
       [问题描述]
       输出：3-5 个关键观点 + 你的建议
   ```

3. 收集所有 subagent 输出
4. 综合分析（controller，不带 persona——中立视角）：
   - 哪些观点一致？
   - 哪些观点冲突？
   - 有没有谁都没想到的角度？
5. 将综合结果呈现给用户做决策

**与 Board Meeting 的区别**：这里 dispatch 的是真正独立的 subagent（真正隔离），
subagent 之间完全不知道对方的存在，真正防止了锚定效应。

## 设计完成后

- 将设计文档保存到合适位置并提交
- 设计自审：扫描占位符、内部矛盾、范围漂移、歧义描述——发现问题直接修
- 让用户审阅设计文档
- 用户确认后调用 `yeaft:writing-plans` 创建实现计划

## 核心原则

- **每次一个问题** — 别一口气问一堆
- **优先选择题** — 比开放问题更容易回答
- **YAGNI** — 狠心砍掉不必要的功能
- **探索替代方案** — 至少提出 2-3 个方案再做选择
- **增量确认** — 呈现设计 → 获得认可 → 继续下一部分
- **灵活调整** — 有什么不对随时返回澄清
