# 测试驱动开发（TDD）

## 概述

先写测试。看它失败。写最小代码通过。

**核心原则：** 如果你没看到测试失败，你就不知道它测的是不是对的东西。

## 铁律

```
没有失败的测试，不写任何生产代码
```

先写了代码再写测试？删掉代码。从头来。

**没有例外：**
- 不要留作"参考"
- 不要"适配"它来写测试
- 不要看它
- 删掉就是删掉

## Red-Green-Refactor

### RED — 写失败的测试

写一个最小测试展示预期行为。

要求：
- 一个行为
- 清晰的名字
- 用真实代码（不用 mock，除非不得已）

### 验证 RED — 看它失败

**必须。不能跳过。**

确认：
- 测试失败（不是报错）
- 失败消息符合预期
- 因为功能缺失而失败（不是拼写错误）

### GREEN — 最小代码

写最简单的代码通过测试。不要多做。

### 验证 GREEN — 看它通过

**必须。**

确认：
- 当前测试通过
- 其他测试也通过
- 输出干净（无错误、警告）

### REFACTOR — 清理

只在 GREEN 之后：
- 消除重复
- 改进命名
- 提取辅助函数

保持测试绿色。不加新行为。

### 重复

下一个行为的下一个失败测试。

## 常见借口

| 借口 | 真相 |
|------|------|
| "太简单不需要测" | 简单代码也会出 bug。测试只需 30 秒 |
| "我之后写测试" | 之后写的测试直接通过，什么都证明不了 |
| "先试试再说" | 第一个修复定下模式。从一开始就做对 |
| "TDD 太教条了" | TDD 就是务实——比调试更快 |
| "手工测试过了" | 临时测试 ≠ 系统测试。没有记录，不能重跑 |
| "删掉 X 小时的工作太浪费" | 沉没成本谬误。保留不可信的代码才是浪费 |

## 红线 — 停下来重头开始

- 先写了代码
- 实现后才写测试
- 测试直接通过了
- 说不清测试为什么失败
- "就这一次例外"

**以上全部意味着：删掉代码。用 TDD 从头来。**

## 验证清单

- [ ] 每个新函数/方法都有测试
- [ ] 看过每个测试失败
- [ ] 每个测试因为预期原因失败
- [ ] 写了最小代码通过
- [ ] 所有测试通过
- [ ] 输出干净
- [ ] 测试用真实代码
- [ ] 边界条件和错误情况已覆盖
