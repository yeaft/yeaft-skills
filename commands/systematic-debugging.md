# 系统调试

## 概述

随机修复浪费时间还制造新 bug。快速补丁掩盖根本问题。

**核心原则：** 在尝试修复之前必须找到根因。治标不治本就是失败。

## 铁律

```
没有根因调查，不允许修复
```

没完成 Phase 1，就不能提出修复方案。

## 四个阶段

每个阶段必须完成后才能进入下一个。

### Phase 1: 根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定重现**
   - 能可靠触发吗？
   - 精确步骤是什么？
   - 每次都发生吗？
   - 不能重现 → 收集更多数据，不要猜

3. **检查最近的改动**
   - 什么改了可能导致这个问题？
   - git diff、最近提交
   - 新依赖、配置改动
   - 环境差异

4. **多组件系统收集证据**
   在每个组件边界添加诊断日志，运行一次收集证据，找到断裂点。

5. **追踪数据流**
   - 坏值从哪里来？
   - 谁用坏值调用了这个？
   - 一直追溯到源头
   - 在源头修复，不在症状处修

### Phase 2: 模式分析

1. **找到工作的例子** — 代码库中类似的能用的代码
2. **对比参考** — 完整阅读参考实现，不要略读
3. **识别差异** — 工作的和不工作的有什么不同？
4. **理解依赖** — 需要什么组件、设置、环境？

### Phase 3: 假设与测试

1. **形成单一假设** — "我认为 X 是根因因为 Y"
2. **最小测试** — 做最小可能的改动来验证假设
3. **一次改一个变量**
4. **验证后再继续**

### Phase 4: 实现修复

1. **创建失败的测试用例** — 使用 `yeaft:tdd` skill
2. **实现单一修复** — 针对根因，一次一个改动
3. **验证** — 测试通过？其他测试没破？
4. **如果 3 次修复失败 → 质疑架构**
   - 每次修复都暴露新的共享状态/耦合？
   - 修复需要"大规模重构"？
   - 和用户讨论，不要继续第 4 次修复尝试

## 红线 — 停下来按流程走

如果你发现自己在想：
- "先快速修一下，之后再调查"
- "试试改 X 看看行不行"
- "同时改多处运行测试"
- "大概是 X，让我修一下"

**以上全部意味着：停。回到 Phase 1。**

## 常见借口

| 借口 | 真相 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快 |
| "紧急情况，没时间走流程" | 系统调试比乱猜更快 |
| "让我先试一下" | 第一次修复定下模式。从一开始就做对 |
| "3 次修复失败了再试一次" | 3+ 次失败 = 架构问题。别再修了，质疑模式 |
