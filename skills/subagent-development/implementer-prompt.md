# Implementer Subagent Prompt 模板

dispatch 实现者 subagent 时使用此模板。

```
Task tool:
  description: "实现任务 N: [任务名称]"
  prompt: |
    <persona>
    [从 Skill tool 加载的 persona 内容——完整粘贴到此处]
    </persona>

    你正在实现任务 N: [任务名称]

    ## 任务描述

    [计划中任务的完整文本——粘贴到此处，不要让 subagent 自己去读文件]

    ## 上下文

    [场景设定：这个任务在整体中的位置、依赖关系、架构上下文]

    ## 开始之前

    如果你对以下方面有疑问：
    - 需求或验收标准
    - 方案或实现策略
    - 依赖或假设
    - 任务描述中任何不清楚的地方

    **现在就问。** 开工前提出所有顾虑。

    ## 你的工作

    需求明确后：
    1. 严格按任务描述实现
    2. 写测试（如果任务要求 TDD 则先写测试）
    3. 验证实现可用
    4. 提交你的工作
    5. 自审（见下文）
    6. 报告结果

    工作目录: [directory]

    **工作中：** 如果遇到意外或不清楚的地方，**提问**。
    随时可以暂停确认。不要猜测或自作假设。

    ## 代码组织

    - 遵循计划定义的文件结构
    - 每个文件一个清晰职责和明确接口
    - 如果文件超出计划预期在膨胀，停下来报告 DONE_WITH_CONCERNS
    - 在现有代码库中遵循既有模式

    ## 超出能力时

    说"这对我来说太难了"完全没问题。差的工作比没有工作更糟。

    **停下来上报当：**
    - 任务需要多个有效方案的架构决策
    - 你需要理解提供范围之外的代码且找不到头绪
    - 你对方案正确性不确定
    - 你读了一个又一个文件但没有进展

    **怎么上报：** 状态报告 BLOCKED 或 NEEDS_CONTEXT。描述你卡在哪里、试了什么、需要什么帮助。

    ## 报告前：自审

    **完整性：** 完整实现了所有需求吗？有遗漏吗？边界条件？
    **质量：** 这是你最好的工作吗？命名清晰准确吗？
    **纪律：** 避免了过度建设（YAGNI）吗？只做了要求的吗？
    **测试：** 测试验证的是行为不是 mock？覆盖全面吗？

    自审发现问题就修，修完再报告。

    ## 报告格式

    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么（或尝试了什么，如果被阻塞）
    - 测试了什么及结果
    - 修改的文件
    - 自审发现（如有）
    - 问题或顾虑
```
