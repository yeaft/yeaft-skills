# 编写实现计划

## 概述

编写全面的实现计划，假设工程师对代码库零上下文。把所有需要知道的都写清楚：每个任务改哪些文件、代码、测试、需要查的文档、怎么验证。把整个计划拆成小任务。DRY、YAGNI、TDD、频繁提交。

假设执行者是熟练开发者，但对我们的工具链和问题域几乎一无所知。假设他们对测试设计不太熟悉。

**开始时宣告：** "我正在使用 writing-plans skill 创建实现计划。"

## 范围检查

如果需求涉及多个独立子系统，应该在 brainstorming 阶段就拆分成子项目。如果没拆，建议拆成多个独立计划——每个子系统一个。每个计划应能独立产出可测试的软件。

## 文件结构

在定义任务之前，先列出将要创建或修改的文件及其职责。这是锁定分解决策的地方。

- 设计清晰边界和明确接口的单元。每个文件一个职责。
- 优先小而聚焦的文件，避免大而全。
- 一起变的文件放在一起。按职责拆分，不按技术层拆。
- 在现有代码库中遵循既有模式。

## 小步任务粒度

**每一步是一个动作（2-5 分钟）：**
- "写失败的测试" — 一步
- "运行确认失败" — 一步
- "写最小代码让测试通过" — 一步
- "运行测试确认通过" — 一步
- "提交" — 一步

## 计划文档头部

**每个计划必须以此开头：**

```markdown
# [功能名称] 实现计划

> **给代理执行者：** 推荐使用 yeaft:subagent-development skill 按任务执行此计划。步骤使用 checkbox (`- [ ]`) 语法跟踪进度。

**目标：** [一句话描述要构建什么]

**架构：** [2-3 句方案概述]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N: [组件名称]

**文件：**
- 创建: `exact/path/to/file.js`
- 修改: `exact/path/to/existing.js:123-145`
- 测试: `test/exact/path/to/test.js`

- [ ] **步骤 1: 写失败的测试**

```javascript
test('specific behavior', () => {
    const result = func(input);
    expect(result).toBe(expected);
});
```

- [ ] **步骤 2: 运行测试确认失败**

运行: `npm test test/path/test.js`
预期: FAIL

- [ ] **步骤 3: 写最小实现**

```javascript
function func(input) {
    return expected;
}
```

- [ ] **步骤 4: 运行测试确认通过**

运行: `npm test test/path/test.js`
预期: PASS

- [ ] **步骤 5: 提交**

```bash
git add test/path/test.js src/path/file.js
git commit -m "feat: add specific feature"
```
````

## 禁止占位符

每一步必须包含工程师需要的实际内容。以下是**计划失败**——永远不要写：
- "TBD"、"TODO"、"后续实现"、"填入细节"
- "添加适当的错误处理" / "添加验证" / "处理边界条件"
- "为上述写测试"（不附带实际测试代码）
- "类似任务 N"（重复写出代码——执行者可能不按顺序读）
- 只描述做什么不展示怎么做的步骤

## 自审

写完完整计划后，用新鲜眼光检查：

1. **需求覆盖：** 逐条扫描需求，能否指向实现它的任务？列出缺口。
2. **占位符扫描：** 搜索上述"禁止占位符"列表中的模式。修复。
3. **类型一致性：** 后面任务用的类型、方法签名、属性名和前面定义的一致吗？

发现问题直接修，不需要重新审阅。

## 执行交接

保存计划后，提供执行选项：

**"计划已完成并保存。两个执行选项：**

**1. Subagent 驱动（推荐）** — 每个任务 dispatch 一个独立 subagent，任务间自动 review

**2. 内联执行** — 在当前 session 中批量执行，带检查点

**选哪个？"**

如果选 Subagent 驱动 → 调用 `yeaft:subagent-development` skill
