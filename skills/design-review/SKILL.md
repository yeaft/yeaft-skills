# 设计审查

用 Dieter Rams 的设计哲学审查 UI/UX —— "好的设计是尽可能少的设计"。

## 流程

dispatch Rams persona subagent：
```
Task tool:
  prompt: |
    <persona>[yeaft:persona-designer-rams 内容]</persona>

    你正在做设计审查。

    审查对象：[UI 截图/设计稿/组件代码/交互描述]

    ## 审查维度

    **1. 第一印象**
    - 用户看到的第一样东西是什么？
    - 3 秒之内能理解这是干什么的吗？
    - 视觉层次清晰吗？

    **2. 减法审计**
    - 哪些元素不挣得它的像素？砍掉。
    - 有没有多余的按钮、文字、装饰？
    - 能不能再少一个元素？
    - "As little design as possible" — 最少的设计就是最好的设计

    **3. 直觉操作**
    - 你妈能不看说明就用吗？
    - 有没有需要思考的操作？如果有，设计失败
    - 交互是否自然、符合预期？

    **4. 边界状态**
    - 空状态看起来怎样？是功能还是 bug？
    - 加载状态、错误状态处理了吗？
    - 文本超长怎么办？内容为零呢？

    **5. 一致性**
    - 和产品其他部分风格一致吗？
    - 间距、字号、颜色有没有偏离设计系统？
    - 交互模式和其他页面一致吗？

    **6. 移动端**
    - 小屏幕适配了吗？
    - 触摸目标够大吗？（最小 44px）
    - 有没有只在桌面端能用的交互？

    输出格式：
    - 判断：✅ 通过 / ⚠️ 需要调整 / ❌ 重做
    - 要砍掉的元素（列表）
    - 要修复的问题（Critical / Important / Minor）
    - 改进建议
```

## 核心原则

- **减法优于加法** — 先考虑能不能砍，再考虑能不能加
- **如果需要说明就是设计失败** — 好设计自己会说话
- **每个像素都要挣得它的位置** — 不挣得就砍掉
- **为信任而设计** — 每个界面决策要么建立信任，要么侵蚀信任

## 什么时候用

- 新 UI 组件或页面完成后
- 交互设计需要评审
- 用户反馈"不好用"或"看不懂"
- 定期 UI 质量检查
