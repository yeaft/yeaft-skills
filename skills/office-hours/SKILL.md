# Office Hours — 需求发现

像创业公司的 office hours 一样——深挖用户真正需要什么，判断技术上是否可行。

## 流程

### Phase 1: 需求发现（Jobs Persona）

dispatch subagent：
```
Task tool:
  prompt: |
    <persona>[yeaft:persona-pm-jobs 内容]</persona>

    你正在做需求发现——帮用户搞清楚他们真正需要什么。

    用户的初始需求：[用户说的内容]
    项目上下文：[当前项目状态]

    你的工作：
    1. 用户说他们要什么？他们实际需要什么？（很少一样）
    2. 这解决了什么痛点？痛点是真实的，还是工程师在自嗨？
    3. 能用一句话描述用户旅程吗？如果不能，太复杂了
    4. MVP 是什么？不是最小功能集——是最小 WOW 体验
    5. 什么是 P0？什么是噪音？

    输出：
    - 核心需求（一句话）
    - 用户痛点分析
    - P0/P1/P2 优先级排列
    - 建议砍掉的功能
    - 需要进一步澄清的问题
```

### Phase 2: 技术可行性（Torvalds Persona）

dispatch subagent：
```
Task tool:
  prompt: |
    <persona>[yeaft:persona-dev-torvalds 内容]</persona>

    你正在做技术可行性判断。

    需求：[Phase 1 的输出]
    项目代码库：[相关代码路径]

    你的工作：
    1. 技术上能不能做？有什么限制？
    2. 最简单的正确方案是什么？
    3. 对现有代码的影响范围？会不会破坏什么？
    4. 估算工作量（小/中/大）
    5. 有没有技术风险或依赖？

    输出：
    - 可行性判断：✅ 可行 / ⚠️ 有风险 / ❌ 不可行
    - 推荐技术方案（一段话）
    - 影响范围
    - 工作量估算
    - 风险和依赖
```

### Phase 3: 综合

收集两个 subagent 的输出，综合呈现：
- 需求分析结果
- 技术可行性结论
- 推荐的下一步
- 需要用户做的决策

## 什么时候用

- 新功能提议
- 用户反馈需要转化为需求
- "这个能做吗" 类问题
- 需要同时考虑产品和技术视角的决策

## 下一步

Office hours 完成后，根据结论：
- 需求明确 + 技术可行 → 调用 `yeaft:brainstorming` 做详细设计
- 需要更多信息 → 列出待澄清问题，等用户回答
- 技术不可行 → 提出替代方案
